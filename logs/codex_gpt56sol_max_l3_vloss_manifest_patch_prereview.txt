## Findings

1. **P1 — The uncommitted sampler incorrectly reuses the loss cap as the ODE denominator, biasing the endpoint.**
   Training is internally consistent as a capped, timestep-weighted x0 loss: the interpolation is \(z_t=t x_0+(1-t)z\) ([flow_schedule.py](/mnt/afs/mogeflow-control/models/raw_motion/flow_schedule.py:91)), and both prediction and target are divided by `max(1-t, 0.05)` ([train_hy273_raw_flow.py](/mnt/afs/mogeflow-control/train_hy273_raw_flow.py:499)). But for \(t>0.95\), this is no longer the physical rectified-flow velocity. The uncommitted sampler forwards `0.05` into the ODE conversion ([sample_hy273_raw.py](/mnt/afs/mogeflow-control/sample_hy273_raw.py:303)) and returns the integrated state rather than the last x0 estimate ([sample_hy273_raw.py](/mnt/afs/mogeflow-control/sample_hy273_raw.py:110)).
   A no-write oracle with perfect constant `x0=1`, initial `z=0` produced:

   - 20 steps: `1.000000`
   - 32 steps: `0.98828125`
   - 64 steps: `0.98476791`
   - 100 steps: `0.983616`

   Thus increasing solver resolution does not converge to x0. Commit `0954577` retained the small numerical sampling floor instead of coupling it to the training cap; the uncommitted change introduces this regression. Separate loss weighting from ODE conversion, and either use the true `1-t` denominator with a numerical floor or stop before one and project to the final x0 prediction.

2. **P1 — “Strict” resume accepts checkpoints missing all three new contract fields.**
   `representation_loss_space`, `velocity_loss_t_eps`, and `source_manifest_sha256` are optional ([train_hy273_raw_flow.py](/mnt/afs/mogeflow-control/train_hy273_raw_flow.py:696)); only the older required tuple is checked for missing fields, while optional fields are compared only if present ([train_hy273_raw_flow.py](/mnt/afs/mogeflow-control/train_hy273_raw_flow.py:719)). A direct probe confirmed that strict resume accepts a checkpoint with all three removed. Present mismatches are correctly rejected.
   `loss_fork` explicitly permits all three mismatches ([train_hy273_raw_flow.py](/mnt/afs/mogeflow-control/train_hy273_raw_flow.py:1345)), but there is no verified parent-manifest/checkpoint lineage. `--resume_sha256` is parsed and copied into trace metadata ([train_hy273_raw_flow.py](/mnt/afs/mogeflow-control/train_hy273_raw_flow.py:217)) but never validates the resume file. Use a versioned checkpoint contract: require the three fields for the new version, reserve legacy acceptance for an explicit mode, and calculate/validate the parent checkpoint SHA for `loss_fork`.

3. **P1 — The proposed one-way evidence chain is not immutable after manifest regeneration.**
   The new builder correctly makes the source manifest source-only ([build_hy273_l3_vloss_source_manifest.py](/mnt/afs/mogeflow-control/tools/build_hy273_l3_vloss_source_manifest.py:12)). The verifier records hashes of the pilot log and traces ([verify_hy273_l3_vloss_preflight.py](/mnt/afs/mogeflow-control/tools/verify_hy273_l3_vloss_preflight.py:203), [verify_hy273_l3_vloss_preflight.py](/mnt/afs/mogeflow-control/tools/verify_hy273_l3_vloss_preflight.py:253)), but production reads only the report and trusts its pass flags ([train_hy273_raw_flow_l3_vloss_scratch_ddp4.sh](/mnt/afs/mogeflow-control/scripts/launch/train_hy273_raw_flow_l3_vloss_scratch_ddp4.sh:91)). It never re-hashes the referenced pilot log/config/traces. Those artifacts can therefore be changed or deleted after report creation without blocking production. A downstream pilot-evidence manifest, checked by production, or a read-only rerun of the verifier is required.

   The source binding also covers only dataset/cache metadata ([build_hy273_l3_vloss_source_manifest.py](/mnt/afs/mogeflow-control/tools/build_hy273_l3_vloss_source_manifest.py:40)), while training reads mutable motion arrays and captions ([kimodo273_datasets.py](/mnt/afs/mogeflow-control/data/kimodo273_datasets.py:141)) and mutable embedding shards ([hytext_cache.py](/mnt/afs/mogeflow-control/models/raw_motion/hytext_cache.py:61)). Changes to those payloads, or to the Python/PyTorch environment, can make pilot evidence stale without changing the manifest.

4. **P1 — The current active manifest/report state is stale and cannot launch the new pilot.**
   `sha256sum --check --strict` reports six mismatches: training, sampling, production launcher, calibration tool, verifier, and model tests. The active manifest also omits the new pilot launcher, manifest builder, sampling tests, and constraint tests, while still hashing downstream pilot/report artifacts ([source manifest](/mnt/afs/mogeflow-control/run_logs/hy273_l3_vloss_source_manifest.sha256:21)). The existing report has neither `source_binding` nor `expected.source_manifest_sha256` ([preflight report](/mnt/afs/mogeflow-control/run_logs/hy273_l3_vloss_jit_preflight_report.json:211)), which production now requires ([production launcher](/mnt/afs/mogeflow-control/scripts/launch/train_hy273_raw_flow_l3_vloss_scratch_ddp4.sh:118)).
   This is fail-closed, which is good, but the current tree is not pilot-ready. After code fixes, rebuild the source-only manifest, run a fresh pilot, create immutable downstream evidence, and regenerate the report.

5. **P2 — Launcher arithmetic is correct, but GPU isolation is only a racy idle check.**
   Production is genuinely pinned to GPU `0,1,2,3`, four ranks, per-rank batch 16, accumulation 2, global batch 128, 200K steps, and 50K saves ([production launcher](/mnt/afs/mogeflow-control/scripts/launch/train_hy273_raw_flow_l3_vloss_scratch_ddp4.sh:13), [production launcher](/mnt/afs/mogeflow-control/scripts/launch/train_hy273_raw_flow_l3_vloss_scratch_ddp4.sh:47)). No trailing overrides are accepted.
   However, the A100/idle check ([production launcher](/mnt/afs/mogeflow-control/scripts/launch/train_hy273_raw_flow_l3_vloss_scratch_ddp4.sh:20)) does not reserve or lock the GPUs before process creation. The pilot verifier checks world size/global batch but ignores the recorded `cuda_visible_devices` value ([verify_hy273_l3_vloss_preflight.py](/mnt/afs/mogeflow-control/tools/verify_hy273_l3_vloss_preflight.py:243)). Add scheduler/lock-based reservation or a final process/UUID recheck immediately before `torchrun`.

6. **P2 — Critical negative and endpoint tests are missing.**
   The new conversion test only asserts local velocity `20` at `t=.99` ([test_raw_flow_model.py](/mnt/afs/mogeflow-control/tests/test_raw_flow_model.py:222)); the sampling test uses two steps, where the `0.05` cap is never active, and checks only finiteness ([test_raw_flow_sampling.py](/mnt/afs/mogeflow-control/tests/test_raw_flow_sampling.py:41)). Resume tests cover present mismatches but not missing fields, `source_manifest_sha256`, or the strict-versus-`loss_fork` matrix ([test_hy273_constraints.py](/mnt/afs/mogeflow-control/tests/test_hy273_constraints.py:118)). Fixed-t calibration stops at exactly `t=.95`, never inside the capped regime ([calibration report](/mnt/afs/mogeflow-control/run_logs/hy273_l3_vloss_jitpm08ps08_calibration_bins_seed3407_n16_alpha0939702.json:1130)). Add oracle endpoint tests across 20/32/64 steps and tamper/staleness integration tests for every evidence-chain edge.
